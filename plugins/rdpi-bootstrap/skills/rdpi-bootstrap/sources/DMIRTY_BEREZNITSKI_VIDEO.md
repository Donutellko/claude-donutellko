Вот структурированная выжимка второго видео — уже не просто пересказ, а **системное понимание процесса**, примерно на ~2 листа A4.

---

# 📌 1. Главный тезис: проблема не в модели — проблема в процессе

Ключевая мысль видео:

> Если ты пишешь в промте “сделай без ошибок” — у тебя нет процесса.

Почему это важно:

* LLM не понимает абстрактные требования типа *“сделай хорошо”*
* это эквивалент плохого ТЗ
* результат становится **лотереей**

👉 Вывод:
**качество = функция процесса, а не модели**

---

# 📌 2. Реальные данные: AI ускоряет… но ухудшает код

## 📊 Исследование (Carnegie Mellon, 2025)

После перехода на AI:

* +500% строк кода в первый месяц
* +55% коммитов
* через 3 месяца → всё возвращается к норме

НО:

* +30% warnings (линтеры)
* +40% сложность кода
* и это НЕ откатывается назад

👉 эффект:
**технический долг накапливается быстрее, чем польза**

---

## 📊 CodeRabbit (2025)

AI-код vs человек:

* +70% больше проблем в PR
* критические ошибки ↑
* security уязвимости:

  * XSS: ×2.7
  * performance: ×8 (!!)

👉 вывод:
AI **ускоряет создание проблем**

---

## 📊 MIT (2026)

Типичный кейс:

* задача на 2 часа → 9 часов с AI
* модель уводит в тупики
* разработчик не может распознать бред

👉 главный риск:
**если ты не понимаешь систему — AI делает хуже**

---

## 📊 Реальный провал (Notebook platform)

* полностью AI-сгенерированная система
* итог:

  * открытая БД
  * 15 млн API ключей
  * приватные данные

👉 идеальный пример:
**AI без процесса = красиво, быстро, опасно**

# 📌 3. Почему компании всё ещё нанимают инженеров

Факты:

* зарплаты: $470k–900k+
* сотни вакансий в топ-компаниях
* massive hiring

👉 парадокс:

> компании, делающие AI, **нанимают больше инженеров**

Почему:

* AI генерирует код
* но **не контролирует архитектуру и качество**

---

# 📌 4. Роль инженера изменилась

Раньше:

* писать код

Теперь:

* управлять AI
* контролировать контекст
* строить архитектуру
* валидировать результат

👉 ключевая трансформация:
**инженер → оркестратор процесса**

---

# 📌 5. Core идея: контекстная инженерия

Формула качества:

👉 качество =
**полнота + корректность – шум – размер контекста**

Важно:

* контекст ограничен
* шум убивает результат
* лишние данные ухудшают вывод

---

# 📌 6. Решение: декомпозиция процесса

Вместо:
❌ “сделай фичу”

Используем pipeline:

1. **Research**
2. **Design**
3. **Plan**
4. **Implement**

👉 каждый этап:

* уменьшает шум
* увеличивает точность
* готовит контекст

---

# 📌 7. Research — сужение реальности

Что происходит:

* агент читает кодовую базу
* находит релевантные части
* создаёт документ

Важно:

* только факты
* без советов
* без “рефакторим тут”

👉 результат:
**сжатая карта системы**

---

# 📌 8. Design — главный этап

Здесь создаётся:

* архитектура (C4)
* data flow
* sequence diagrams
* API контракты
* тест-стратегия
* риски (ADR)

👉 ключевая идея:

> ошибки здесь дешевле, чем в коде

---

💡 инсайт:
раньше:

* код = основная работа

теперь:

* **дизайн = основная работа**

---

# 📌 9. Plan — превращение дизайна в шаги

Ошибка новичков:

* один огромный план

Правильно:

* разбить на **фазы**
* каждая фаза:

  * завершённая
  * проверяемая
  * коммитится отдельно

👉 результат:
**контролируемая имплементация**

---

# 📌 10. Implement — уже не главное

Факт:

> код пишется только на одном этапе

И это:

* самый простой этап
* почти механический

Если:

* дизайн хороший
* план точный

---

# 📌 11. Ключевая идея: контекст передаётся по цепочке

```
Research → Design → Plan → Implementation
```

Каждый этап:

* создаёт контекст
* передаёт дальше

👉 итог:
**модель не “думает” — она следует структуре**

---

# 📌 12. Multi-agent подход

Вместо одного агента:

👥 команда:

* coder
* reviewer
* security
* tester
* architect

Правила:

* никто не делает чужую работу
* чёткое разделение ролей

👉 как в реальной команде

---

# 📌 13. Quality gates (критично)

Каждая фаза проверяется:

* билд
* тесты
* линтеры
* сложность
* соответствие архитектуре
* безопасность

👉 если не прошёл:
❌ не идём дальше

---

# 📌 14. Контекст = инженерный артефакт

Это не:

* 5 строк промта

Это:

* research doc
* дизайн
* план
* стандарты
* предыдущие результаты

👉 именно это даёт качество

---

# 📌 15. Почему “универсальные промты” не работают

Причины:

* разный стек
* разная архитектура
* разные стандарты

👉 вывод:

> промты — это **производная от процесса**

---

# 📌 16. Специализация — ключ к качеству

Разные задачи → разные промты:

* feature vs bug
* backend vs frontend
* разные языки

👉 один промт:
❌ “сделай фичу” → плохой результат

---

# 📌 17. Главный практический инсайт

> Ты не должен объяснять системе код —
> система должна сама его исследовать

---

# 📌 18. Почему этот подход работает

Он:

* уменьшает контекст
* убирает шум
* фиксирует решения заранее
* делает код предсказуемым

👉 результат:
**контролируемая генерация**

---

# 📌 19. Основная ошибка разработчиков

Текущий подход:

```
prompt → код → фикс → повтор
```

Это:

* хаос
* нестабильность
* деградация

---

Правильный подход:

```
research → design → plan → code → review
```

---

# 📌 20. Финальные выводы

* AI без процесса = техдолг
* скорость без контроля = деградация
* главный навык — не писать код
* а **строить процесс вокруг AI**

---

👉 Не пиши “сделай хорошо” — строй систему
👉 Не давай весь код — давай релевантный контекст
👉 Не генерируй код сразу — сначала дизайн
👉 Не доверяй модели — ставь гейты качества
👉 Не будь кодером — будь архитектором процесса
