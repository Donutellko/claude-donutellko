# 🧠 Полный workflow (Dex Horthy, 2026)

## 🔹 0. (Неявно) Ticket / Intent

* исходная задача (тикет, идея, feature)
* **важно:** дальше она *скрывается* от части пайплайна (для чистоты ресёрча)

---

## 1. ❓ Questions (генерация вопросов)

👉 *новый этап, которого не было в классическом RPI*

* модель НЕ идёт сразу читать код
* сначала формирует:

  * какие части системы важны
  * какие вопросы нужно задать к кодовой базе

📌 ключевая идея:

> не «исследуй это», а «сформируй, ЧТО исследовать»

---

## 2. 🔍 Research (исследование кода)

* отдельный context window (!)
* **без знания задачи**
* только факты:

  * как устроена система
  * какие есть зависимости
  * какие паттерны используются

📌 принцип:

* ❌ без мнений
* ❌ без предложений
* ✅ только «как есть сейчас»

---

## 3. 🧠 Design (дизайн / концепция)

👉 один из самых важных новых этапов

* формируется **design doc (~200 строк)**
* включает:

  * текущее состояние
  * желаемое состояние
  * найденные паттерны
  * предлагаемый подход
  * открытые вопросы

📌 цель:

> синхронизация человека и агента ДО кода

💡 это:

* замена архитектурного митинга
* точка для "brain surgery" над агентом

---

## 4. 🧩 Structure / Outline (структура)

* разбивка на этапы (фазы реализации)
* **без кода**
* high-level план:

  * порядок действий
  * как тестировать
  * checkpoints

📌 ключевая идея:

> избегаем "horizontal plans" (сначала весь backend, потом frontend)

вместо этого:
👉 **vertical slicing**

* куски функциональности end-to-end

---

## 5. 📋 Plan (детальный план)

* уже конкретные изменения:

  * какие файлы
  * какие функции
  * какие изменения

📌 но:

* **больше не читается детально**
* используется как:

  * инструкция для агента
  * не как артефакт для ревью

---

## 6. 🌿 Worktree / Work setup

* подготовка окружения:

  * ветка
  * изоляция изменений
  * возможно sandbox

(в докладе почти не раскрыт, но упомянут как этап)

---

## 7. ⚙️ Implement (реализация)

* генерация кода агентом
* **с проверками по ходу (vertical checkpoints)**

📌 ключевой сдвиг:

> НЕ доверять полностью → проверять частями

---

## 8. 🔁 PR (Pull Request)

* финальный результат
* ревью уже:

  * по коду (не по плану!)
  * с меньшим количеством сюрпризов

---

# 🔁 Сравнение с классическим RPI

### Было:

```
Research → Plan → Implement
```

### Стало (CRISPY):

```
Questions → Research → Design → Structure → Plan → Worktree → Implement → PR
```

# 💡 Главные идеи Декса (очень важно)

## 1. ❌ Нет "magic prompt"

* если нужно говорить "magic words" → система сломана

---

## 2. 🧠 Не аутсорсить мышление

> AI — это усилитель, не замена

---

## 3. 📖 Читать код (обновлённая позиция)

* раньше: "не читайте код"
* теперь:

  > "Пожалуйста, читайте код"

---

## 4. ⚖️ Instruction budget

* ~150–200 инструкций максимум
* большие промпты → деградация

👉 решение:

* разбивать на этапы (pipeline)

---

## 5. 🧱 Контроль через pipeline, а не prompt

> ❌ prompt как workflow
> ✅ код как workflow

---

## 6. 📉 Минимизировать context window

* меньше контекста → лучше качество
* избегать "dumb zone"

---

## 7. 🔄 Vertical development

вместо:

```
DB → backend → API → frontend
```

делать:

```
endpoint → UI → service → DB → тест
```

---

## 8. ⚡ Реалистичный буст

* не 10x
* а:
  👉 **2–3x без деградации качества**
