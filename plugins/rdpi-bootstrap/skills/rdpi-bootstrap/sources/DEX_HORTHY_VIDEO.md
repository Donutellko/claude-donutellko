Вот подробная выжимка ключевых идей из видео
**“Advanced Context Engineering for Coding Agents” (Dex Horthy)** — примерно на объём ~2 листов A4, структурированно и без воды.

---

# 📌 1. Главная идея: проблема не в AI, а в том, как мы им пользуемся

Основной тезис:

> AI увеличивает количество кода, но НЕ обязательно увеличивает прогресс.

На практике:

* код генерируется быстрее
* но **переработок (churn) становится ещё больше**
* особенно плохо это работает в **старых (brownfield) проектах** ([DEV Community][1])

👉 Итог: без правильного подхода AI превращается в **генератор техдолга**.

---

# 📌 2. Единственный рычаг управления — это контекст

LLM:

* не имеют памяти между сессиями
* недетерминированы
* полностью зависят от текущего контекста

👉 Ключевая формула:

> “Better tokens in → better tokens out” ([DEV Community][1])

Важно:

* **больше контекста ≠ лучше результат**
* качество важнее количества

---

# 📌 3. “Dumb zone” — критическая проблема

Когда контекст становится слишком большим, модель:

* начинает хуже думать
* чаще ошибается
* становится нестабильной

📉 Обычно это происходит уже при ~40% окна контекста ([DEV Community][1])

Причины:

* большие JSON / логи
* дампы файлов
* длинные истории чата
* повторяющиеся исправления

👉 Вывод:
**перегруженный контекст убивает качество, даже у сильных моделей**

---

# 📌 4. Траектория диалога влияет на результат

LLM учится по ходу диалога.

Если паттерн:

1. модель ошиблась
2. пользователь ругает
3. модель снова ошибается

👉 модель продолжит ошибаться

Это называется плохая **trajectory**.

Вывод:

* иногда **лучше начать заново**, чем “чинить” диалог

---

# 📌 5. Intentional Compaction — ключевая техника

Это центральная идея видео.

👉 Суть:
вместо накопления контекста — **сжимать его в качественное резюме**

Как делать:

1. собрать знания
2. сжать в структурированный markdown
3. проверить вручную
4. начать новую сессию с этим контекстом

Что включать:

* важные файлы и куски кода
* архитектурные решения
* ограничения

Что НЕ включать:

* логи
* трейсинг
* мусор

👉 Это превращает хаос в **контролируемую систему**

---

# 📌 6. Sub-agents — не про роли, а про контекст

Распространённая ошибка:
думать, что агенты = “frontend”, “backend” и т.д.

На самом деле:
👉 sub-agent = способ открыть новый чистый контекст

Пример:

* агент читает весь репозиторий
* возвращает краткое summary
* основной агент работает только с этим summary

👉 Это позволяет:

* масштабировать задачи
* не попадать в dumb zone

---

# 📌 7. Главный workflow: RPI (Research → Plan → Implement)

Это ядро всей методологии.

---

## 🔍 Research (исследование)

Цель:

* понять, как система реально работает

Правила:

* читать код, а не документацию
* фиксировать выводы
* проверять вручную

👉 если контекст неправильный → модель начинает выдумывать

---

## 📋 Plan (планирование)

Самая важная стадия.

Хороший план:

* конкретные шаги
* ссылки на файлы
* проверки после каждого шага
* явные риски

👉 плохой план → десятки плохих строк кода
👉 плохое исследование → сотни

---

## ⚙️ Implement (реализация)

Если план хороший:

* выполнение становится механическим
* ошибок меньше
* контекст маленький

👉 именно здесь AI реально эффективен

---

# 📌 8. Code review меняется

С AI нельзя просто читать код:

* слишком большой объём

Новая стратегия:

* ревью **исследования и планов**
* проверка логики, а не синтаксиса

👉 это сохраняет архитектуру при росте скорости разработки

---

# 📌 9. AI не заменяет мышление

Ключевая мысль:

> AI усиливает мышление, но не заменяет его ([DEV Community][1])

Особенно:

* архитектура
* сложные рефакторинги
* legacy-системы

👉 тут всегда нужен человек

---

# 📌 10. Выбор подхода по сложности задачи

| Тип задачи                     | Подход          |
| ------------------------------ | --------------- |
| UI-правка                      | прямой prompt   |
| маленькая фича                 | лёгкий план     |
| изменения в нескольких модулях | research + plan |
| сложный рефакторинг            | полный RPI      |

👉 чем сложнее задача — тем больше нужно контроля контекста

---

# 📌 11. Главный сдвиг: инженер → оркестратор

Будущее разработчика:

* управлять агентами
* контролировать контекст
* проверять результаты

👉 ценность смещается:
**от написания кода → к управлению процессом**

---

# 📌 12. Главные выводы (сжато)

* Контекст — единственное, что реально контролирует LLM
* Больше данных часто ухудшает результат
* “Dumb zone” — главный враг
* Нужно регулярно **сжимать контекст**
* Планирование важнее кодинга
* Sub-agents — способ управлять контекстом
* AI увеличивает скорость, но может убить качество
* Мышление остаётся главным bottleneck

---

# 🧠 Коротко (если совсем сжать)

👉 Не давай модели больше — давай ей лучше
👉 Не исправляй диалог — пересобирай контекст
👉 Не проси код — сначала сделай план
👉 Не масштабируй код — масштабируй мышление
